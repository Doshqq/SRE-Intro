### 8. DNS service discovery explanation

**Answer:** The gateway finds the events service using Docker's embedded DNS server. The DNS server runs at `127.0.0.11` (as shown in `/etc/resolv.conf` inside the container). When the gateway resolves the hostname `events`, Docker's DNS server returns the IP address `172.18.0.5` (the events container's IP on the app_default network). This service discovery happens automatically because all containers are on the same Docker Compose network, and Docker maintains a mapping of service names to container IPs.

## Bonus Task — Trace a Request Across Services (2 pts)

### Full timestamped logs

```bash
$ docker compose down && docker compose up -d
# Wait 5 seconds...

$ RES=$(curl -s -X POST http://localhost:3080/events/1/reserve \
  -H "Content-Type: application/json" -d '{"quantity":1}')
$ RES_ID=$(echo "$RES" | python3 -c "import sys,json; print(json.load(sys.stdin)['reservation_id'])")
$ curl -s -X POST "http://localhost:3080/reserve/$RES_ID/pay"

$ docker compose logs --timestamps
```

### Annotated log lines

**Request flow for reservation ID: aacdc2b0-aad0-437f-9655-2a784b99de11**

1. **Gateway receives reserve request** (11:16:54.321)
   - `gateway-1 | 2026-06-11T11:16:54.321479003Z {"time":"2026-06-11 11:16:54,321","level":"INFO","service":"gateway","msg":"HTTP Request: POST http://events:8081/events/1/reserve "HTTP/1.1 200 OK"}`
   - Gateway receives POST /events/1/reserve from client

2. **Events processes reserve** (11:16:54.318)
   - `events-1 | 2026-06-11T11:16:54.319460337Z {"time":"2026-06-11 11:16:54,318","level":"INFO","service":"events","msg":"Reserved 1 tickets for event 1: aacdc2b0-aad0-437f-9655-2a784b99de11"}`
   - Events creates reservation and returns 200 OK
   - **Time from gateway to events: ~3ms**

3. **Gateway returns reserve response** (11:16:54.322)
   - `gateway-1 | 2026-06-11T11:16:54.322503878Z INFO:     192.168.65.1:23134 - "POST /events/1/reserve HTTP/1.1" 200 OK`
   - Gateway returns reservation to client

4. **Gateway receives pay request** (11:17:09.247)
   - `gateway-1 | 2026-06-11T11:17:09.269002927Z {"time":"2026-06-11 11:17:09,247","level":"INFO","service":"gateway","msg":"HTTP Request: POST http://payments:8082/charge "HTTP/1.1 200 OK"}`
   - Gateway receives POST /reserve/{id}/pay from client (14.9 seconds after reserve)

5. **Payments processes charge** (11:17:09.237)
   - `payments-1 | 2026-06-11T11:17:09.241141469Z {"time":"2026-06-11 11:17:09,237","level":"INFO","service":"payments","msg":"Payment success: PAY-1276BCD9 for aacdc2b0-aad0-437f-9655-2a784b99de11"}`
   - Payments processes charge and returns 200 OK
   - **Time from gateway to payments: ~10ms**

6. **Gateway confirms reservation** (11:17:09.309)
   - `gateway-1 | 2026-06-11T11:17:09.310107010Z {"time":"2026-06-11 11:17:09,309","level":"INFO","service":"gateway","msg":"HTTP Request: POST http://events:8081/reservations/aacdc2b0-aad0-437f-9655-2a784b99de11/confirm "HTTP/1.1 200 OK"}`
   - Gateway forwards confirmation request to events

7. **Events confirms order** (11:17:09.307)
   - `events-1 | 2026-06-11T11:17:09.308444635Z {"time":"2026-06-11 11:17:09,307","level":"INFO","service":"events","msg":"Order confirmed: aacdc2b0-aad0-437f-9655-2a784b99de11"}`
   - Events confirms order and returns 200 OK
   - **Time from gateway to events (confirm): ~2ms**

8. **Gateway returns final response** (11:17:09.313)
   - `gateway-1 | 2026-06-11T11:17:09.313106052Z INFO:     192.168.65.1:65236 - "POST /reserve/aacdc2b0-aad0-437f-9655-2a784b99de11/pay HTTP/1.1" 200 OK`
   - Gateway returns confirmed order to client

### End-to-end time calculation

**Answer:** The total end-to-end time from the gateway receiving the pay request to returning the final response is approximately **66ms** (from 11:17:09.247 to 11:17:09.313). The request flows through: gateway → payments (10ms) → gateway → events (2ms) → gateway → client. The actual service-to-service communication is very fast (sub-10ms), with the majority of time being the 14.9 second gap between the user making the reserve request and the pay request (manual delay).
